# Auto-Update Guide

This document explains how automatic package updates work in this dotfiles repository.

## Overview

Packages in `nix/` are automatically checked for upstream updates on a weekly schedule
using [nvfetcher](https://github.com/berberman/nvfetcher). If new versions are found
and all packages build successfully, the updated sources are committed to `main`.

## How it works

```
(nvfetcher.toml)                   (nix/_sources/)
┌──────────────┐                   ┌──────────────────┐
│ hanabi       │─── nvfetcher ────→│ _generated.json  │
│ tlauncher    │    (weekly)       │ _generated.nix   │
│ codegraph    │                   └────────┬─────────┘
│ 9router      │                            │
│ openhuman    │                    ┌────────▼─────────┐
└──────────────┘                    │ _default.nix     │
                                    │ (reads JSON →    │
                                    │  builds Nix src) │
                                    └────────┬─────────┘
                                             │
                          ┌──────────────────▼──────────┐
                          │ nix/hanabi.nix              │
                          │ nix/tlauncher.nix           │
                          │   import ./_sources/default │
                          │   inherit (sources) src ver │
                          └─────────────────────────────┘
```

1. **nvfetcher** checks upstream sources (GitHub releases, git commits, web pages)
2. Detects new versions and recomputes SRI hashes
3. Generates `_generated.json` and `_generated.nix` with updated source info
4. `_default.nix` reads the JSON and constructs `fetchFromGitHub`/`fetchurl` calls
5. Package derivations (`hanabi.nix`, `tlauncher.nix`) inherit `version` and `src` from sources
6. All packages are built to verify nothing broke
7. If all builds pass, changes are committed to `main`

## File structure

| File | Role |
|---|---|
| `nvfetcher.toml` | Package source configuration. All packages in `nix/` have an entry here. |
| `nix/_sources/_generated.json` | Auto-generated by nvfetcher. Contains version, hash, and fetch metadata for each package. |
| `nix/_sources/_generated.nix` | Auto-generated Nix expression (not used directly — the JSON is the source of truth). |
| `nix/_sources/_default.nix` | Helper that reads `_generated.json` and constructs Nix fetcher calls (`fetchFromGitHub`, `fetchurl`). |
| `.github/workflows/update.yml` | Scheduled workflow: runs nvfetcher → builds all packages → commits on success. |
| `.github/workflows/build.yml` | Build workflow: triggered on push to `main`. Builds + signs + pushes to R2 cache. |

## Adding a new package to auto-update

1. **Create the package derivation** in `nix/<name>.nix` (if not already there).
2. **Add an entry** to `nvfetcher.toml`:

```toml
[my-package]
src.github = "owner/my-package"
fetch.github = "owner/my-package"
```

3. **Modify the derivation** to source `version` and `src` from the generated sources:

```nix
{ ... }:

let
  mkMyPackage =
    pkgs:
    let
      sources = import ./_sources/_default.nix { inherit pkgs; };
      myPkg = sources.my-package;
    in
    pkgs.callPackage (
      { lib, stdenv, fetchFromGitHub }:
      stdenv.mkDerivation {
        pname = "my-package";
        inherit (myPkg) version src;
        ...
      }
    ) { };
in
{
  perSystem = { pkgs, ... }: {
    packages.my-package = mkMyPackage pkgs;
  };
}
```

4. **Regenerate sources**:

```bash
nix shell nixpkgs#nvfetcher -c nvfetcher --build-dir nix/_sources
mv nix/_sources/generated.nix nix/_sources/_generated.nix
mv nix/_sources/generated.json nix/_sources/_generated.json
```

5. **Verify** the package builds:

```bash
nix build .#packages.x86_64-linux.my-package --no-link
```

## Ignoring a package (pinning)

Packages that should NOT be auto-updated get `pinned = true` in `nvfetcher.toml`:

```toml
[codegraph]
src.github = "colbymchenry/codegraph"
fetch.github = "colbymchenry/codegraph"
pinned = true
```

Pinned packages are checked for a version only on the first run (to establish a baseline).
On subsequent weekly runs, nvfetcher skips them entirely.

Pinned packages remain in `_generated.json` but never trigger an update commit.

## nvfetcher source types

### GitHub releases (track latest release)

```toml
[foo]
src.github = "owner/repo"
fetch.github = "owner/repo"
```

### GitHub tags (track latest tag)

```toml
[foo]
src.github_tag = "owner/repo"
fetch.github = "owner/repo"
```

### Git commit (track latest commit on default branch)

```toml
[foo]
src.git = "https://github.com/owner/repo.git"
fetch.github = "owner/repo"
```

### Web scraping (regex from HTML)

```toml
[foo]
src.webpage = "https://example.com/download"
src.regex = 'Product ([\d.]+)'
fetch.url = "https://example.com/download/latest"
```

### HTTP header (extract version from response headers)

```toml
[foo]
src.httpheader = "https://example.com/download"
src.regex = 'location: .*/([\d.]+)/'
fetch.url = "https://example.com/download/latest"
```

### Direct URL (static URL, manual version)

```toml
[foo]
src.manual = "1.0"
fetch.url = "https://example.com/download/foo-$ver.tar.gz"
```

### PyPI

```toml
[foo]
src.pypi = "package-name"
fetch.pypi = "package-name"
```

## The update workflow

File: `.github/workflows/update.yml`

```yaml
on:
  schedule:
    - cron: '0 6 * * 1'   # Every Monday at 06:00 UTC
  workflow_dispatch:        # Manual trigger from any branch
```

Steps:

1. **Checkout** the repository
2. **Install Nix** via `DeterminateSystems/nix-installer-action`
3. **Run nvfetcher** to check all sources and regenerate `generated.nix`/`generated.json`
4. **Rename** generated files (`generated.*` → `_generated.*`)
5. **Build verification** — dynamically discovers all packages via `nix eval` and builds them in parallel. If any single package fails to build, the entire step fails
6. **Commit** — only runs if all previous steps succeeded. Uses `git-auto-commit-action` to push the updated `nix/_sources/` directory

### On build failure

When the build verification step fails:

- The workflow stops immediately (`exit 1` on the first failed build)
- The `git-auto-commit` step is skipped
- No commit is made — `main` stays on the working version
- A GitHub notification is sent (if configured)
- Manual investigation is needed: check the workflow logs to see which package failed and why

Since scheduled runs only happen weekly, a failed run won't spam notifications. The next
scheduled run will try again with whatever versions are current at that time.

### On success

When all builds pass:

- Updated `nix/_sources/_generated.json` and `nix/_sources/_generated.nix` are committed
- The push to `main` triggers `build.yml` automatically (via `paths: nix/**`)
- `build.yml` builds all packages, signs them, and pushes to the R2 binary cache
- All consumers get updated packages on their next rebuild

## Manual run

Go to **Actions → Auto-Update Nix Packages → Run workflow** — you can select any branch.

## Extending to new packages

Even if a package should not auto-update (pinned), add it to `nvfetcher.toml` anyway.
This keeps the source config centralized. The `_default.nix` helper only applies to
packages that actually import it — pinned packages that haven't been migrated to use
the generated sources are unaffected.
